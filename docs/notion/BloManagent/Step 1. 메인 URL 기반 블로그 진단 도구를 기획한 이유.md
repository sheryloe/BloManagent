# Step 1. 메인 URL 기반 블로그 진단 도구를 기획한 이유

## SEO 패키지

- SEO 제목: 메인 URL 하나로 시작하는 블로그 진단 도구를 만든 이유
- 슬러그: `main-url-blog-diagnostics-tool-idea`
- 메타 설명: 메인 URL만 입력하면 공개 블로그를 수집하고 게시글별 qualityScore를 계산하는 BloManagent를 왜 기획했는지, 문제 정의부터 로컬 중심 워크플로우까지 정리합니다.
- 핵심 키워드: 메인 URL 블로그 분석, 블로그 진단 도구, 로컬 블로그 대시보드
- 보조 키워드: 티스토리 블로그 분석, 워드프레스 글 진단, qualityScore 설계
- 추천 카테고리: 개발, 사이드프로젝트, 블로그운영
- 추천 태그: `블로그분석`, `사이드프로젝트`, `Tistory`, `WordPress`, `qualityScore`

## 도입

블로그를 운영하다 보면 새 글을 쓰는 것보다 더 어려운 순간이 있습니다.  
바로 “지금 내 블로그에서 먼저 손봐야 할 글이 무엇인가”를 판단하는 순간입니다.

문장 교정 도구나 생성형 AI는 많습니다. 하지만 대부분은 개별 글 작성에 집중되어 있습니다. 실제 운영자 입장에서는 아래 질문이 더 먼저 나옵니다.

- 이 블로그에서 점수가 가장 낮은 글은 무엇인가
- 어떤 문제가 반복되는가
- 티스토리와 워드프레스를 같은 기준으로 볼 수 있는가
- API 키 없이 공개 글만으로도 기본 진단이 가능한가

BloManagent는 이 질문에서 출발한 프로젝트입니다.

## 이 글에서 다루는 내용

1. 왜 메인 URL 하나로 시작해야 했는가
2. 왜 로컬 중심 워크플로우가 맞았는가
3. 왜 블로그 평균보다 게시글별 진단을 먼저 보여주기로 했는가
4. 첫 번째 제품 원칙을 어떻게 정했는가

## 왜 메인 URL 하나로 시작해야 했나

현실에서는 사용자가 RSS URL이나 관리자 API 엔드포인트를 기억하고 있지 않은 경우가 많습니다.  
대부분은 `https://example.tistory.com` 같은 메인 주소만 알고 있습니다.

그래서 제품의 첫 입력은 복잡하면 안 됐습니다.

- 메인 URL만 받는다
- RSS, sitemap, wp-json은 내부에서 탐색한다
- 공개 데이터만으로 수집 가능한 범위를 먼저 보여준다

이렇게 해야 등록 장벽이 낮아집니다.  
도구를 한 번 열고 끝나는 것이 아니라, 블로그 운영 중 계속 꺼내 쓰는 워크스페이스가 되려면 첫 단계가 가장 단순해야 했습니다.

## 왜 로컬 중심 워크플로우가 맞았나

처음부터 SaaS로 가는 선택지도 있었습니다. 하지만 이 프로젝트는 로컬 중심이 더 맞았습니다.

### 1. 공개 글 분석은 개인 워크플로우에 가깝다

사용자는 대개 여러 블로그를 본인 컴퓨터에서 관리합니다.  
“지금 한 번만 분석하고 싶다”는 니즈가 더 강하지, 항상 켜져 있는 서버를 원하지는 않습니다.

### 2. 설치와 실행이 단순해야 한다

SQLite 하나로 데이터를 들고 가면 백업도 쉽고, Windows 환경에서도 관리가 쉽습니다.  
나중에 Electron이나 Tauri로 묶기도 편합니다.

### 3. API 키 의존성을 최소화하고 싶었다

처음부터 OpenAI나 Google API 키가 있어야만 동작하는 구조는 진입 장벽이 높습니다.  
그래서 기본 분석 엔진을 알고리즘으로 두고, AI는 선택형 보강 기능으로만 두기로 했습니다.

## 왜 블로그 평균이 아니라 게시글별 진단을 우선했나

블로그 전체 평균 점수는 보기에는 편합니다. 하지만 액션으로 연결되기 어렵습니다.

예를 들어 평균이 71점이라고 해도 사용자가 바로 할 수 있는 일은 많지 않습니다.  
반대로 아래 정보는 즉시 행동으로 이어집니다.

- 가장 낮은 qualityScore 글 3개
- 반복 이슈 상위 3개
- 각 글의 개선 포인트 2개

그래서 BloManagent는 “블로그 상태를 알려주는 지표”보다 “지금 수정할 수 있는 글 목록”을 먼저 보여주는 방향을 택했습니다.

## 이 프로젝트의 첫 번째 원칙

기획 단계에서 아래 원칙을 먼저 고정했습니다.

### 원칙 1. 메인 URL만 받아도 돌아가야 한다

고급 옵션은 있더라도, 기본 흐름은 URL 하나로 끝나야 했습니다.

### 원칙 2. 점수는 설명 가능해야 한다

qualityScore는 반드시 세부 항목으로 쪼개져야 했고, 사용자가 “왜 이 점수인지”를 이해할 수 있어야 했습니다.

### 원칙 3. 공개 데이터만 다룬다

로그인 쿠키, 비공개 글, 관리자 통계는 범위에서 제외했습니다.  
이 원칙이 있어야 정책과 운영 범위를 선명하게 유지할 수 있습니다.

### 원칙 4. 로컬에서 바로 돌릴 수 있어야 한다

오픈소스로 열어두고, 원하는 사람이 자신의 환경에서 쉽게 실행할 수 있어야 했습니다.

## 마무리

Step 1에서는 기능을 많이 넣는 것보다, 어떤 질문에 답하는 도구인지 먼저 고정하는 일이 더 중요했습니다.  
BloManagent는 “공개 블로그를 메인 URL 하나로 수집하고, 게시글별로 바로 수정할 수 있는 진단을 준다”는 목표에서 시작했습니다.
