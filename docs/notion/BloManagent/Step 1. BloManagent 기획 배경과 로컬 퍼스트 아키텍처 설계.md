# BloManagent 기획 배경과 로컬 퍼스트 아키텍처 설계

> 메타 설명  
> 티스토리, 네이버 블로그, Blogger를 메인 URL 기준으로 수집하고 분석하는 로컬 퍼스트 블로그 대시보드 BloManagent의 기획 배경과 전체 아키텍처 설계 과정을 정리합니다.

> 추천 슬러그  
> `blomanagent-local-first-blog-dashboard-architecture`

> 핵심 키워드  
> 로컬 퍼스트 블로그 분석, 티스토리 블로그 분석 도구, 네이버 블로그 대시보드, 블로그 아키텍처 설계

## 들어가며

블로그를 운영하다 보면 가장 먼저 부딪히는 문제는 "지금 잘하고 있는지"를 객관적으로 보기 어렵다는 점입니다.  
개별 글을 요약하거나 문장을 교정하는 도구는 많지만, 블로그 전체 방향을 시간의 흐름에 따라 비교해 주는 도구는 생각보다 드뭅니다.

그래서 이번 프로젝트에서는 **BloManagent** 라는 로컬 퍼스트 블로그 분석 대시보드를 설계했습니다.  
핵심 목표는 단순합니다. **블로그 메인 URL 하나만 입력하면, 글을 수집하고, 분석 이력을 쌓고, 다음 운영 방향까지 보여주는 것**입니다.

## 왜 이 프로젝트가 필요했는가

실제 블로그 운영에서는 다음과 같은 문제가 자주 발생합니다.

- 지난달보다 콘텐츠 품질이 좋아졌는지 숫자로 확인하기 어렵다
- 티스토리, 네이버 블로그, Blogger를 각각 다른 방식으로 봐야 한다
- 외부 공식 점수 API가 없어서 공통 기준이 없다
- 항상 켜져 있는 서버 없이도 필요할 때만 실행하고 싶다

이 문제를 해결하려면 일회성 요약기가 아니라, **히스토리를 쌓는 블로그 운영 대시보드**가 필요했습니다.

## 문제 정의

BloManagent는 다음 문제를 해결하는 데 초점을 맞췄습니다.

### 1. 블로그 메인 URL만으로 시작해야 한다

사용자는 포스트 URL을 일일이 넣고 싶지 않습니다.  
대부분은 `https://example.tistory.com` 같은 메인 주소만 알고 있습니다.  
그래서 시스템은 메인 URL을 시작점으로 삼아 포스트 링크를 자동으로 발견해야 합니다.

### 2. 과거 분석을 덮어쓰면 안 된다

블로그는 시간이 지날수록 바뀝니다.

- 댓글이 늘어날 수 있고
- 공감 수가 바뀔 수 있고
- 운영 전략도 달라질 수 있습니다

따라서 같은 글이라도 **2026년 3월 분석 결과와 2026년 6월 분석 결과는 분리되어야 합니다.**  
그래서 분석 결과는 스냅샷으로 쌓는 구조가 필수였습니다.

### 3. 수동 실행 중심이어야 한다

이 프로젝트는 상시 서비스가 아니라 **필요할 때 직접 켜서 쓰는 로컬 앱**을 목표로 했습니다.  
그래서 자동 스케줄러보다 `Analyze Now` 버튼 기반 UX를 우선 설계했습니다.

## 핵심 제품 목표

BloManagent의 핵심 목표는 다음과 같습니다.

- 블로그 메인 URL 등록
- 플랫폼 자동 감지
- 포스트 링크 자동 수집
- SQLite DB에 포스트 및 분석 이력 저장
- AI 제공자 선택형 분석
- 주간 변화 비교
- 다음 운영 액션 추천

## 왜 로컬 퍼스트 구조를 선택했는가

이번 프로젝트에서 로컬 퍼스트는 단순한 취향이 아니라 요구사항 그 자체였습니다.

### 첫째, 실행 부담이 낮아야 한다

사용자는 필요할 때만 앱을 켜고 분석하고 싶어 합니다.  
클라우드 서버, 배포, 인증, 백그라운드 워커는 1차 목표가 아닙니다.

### 둘째, 데이터가 로컬에 남아야 한다

블로그 포스트 원문, 분석 결과, 비용 로그, 추천 기록은 사용자의 자산입니다.  
SQLite를 쓰면 설치와 백업도 단순하고, Windows 환경에서 다루기도 쉽습니다.

### 셋째, 향후 데스크톱 패키징이 쉬워야 한다

처음에는 로컬 웹앱으로 시작하되, 나중에 Electron이나 Tauri 기반 `.exe` 로 패키징하기 쉽게 구조를 나누는 것이 중요했습니다.

## 전체 아키텍처 구성

아키텍처는 크게 세 층으로 나눴습니다.

### 1. Web UI

React + Vite + React Router로 구성했습니다.  
대시보드, 블로그 목록, 블로그 상세, 리포트, 설정, 로그 페이지를 빠르게 분리할 수 있고, 나중에 Electron/Tauri에 넣기도 편한 조합입니다.

### 2. Local API / Orchestrator

Fastify 서버가 다음 역할을 담당합니다.

- 블로그 등록 API
- 포스트 수집 실행
- 분석 런 생성
- SQLite 저장
- 제공자 모델 조회
- 설정 관리

### 3. Shared Contracts

`packages/shared`에 타입과 Zod 스키마를 모아, 프론트와 백엔드가 같은 계약을 사용하도록 만들었습니다.  
이 구조를 두면 나중에 기능이 커져도 DTO가 쉽게 무너지지 않습니다.

## 정리

Step 1에서는 "무엇을 만들 것인가"보다 **왜 이렇게 만들어야 하는가**를 먼저 고정했습니다.  
이 프로젝트는 블로그 글 요약기가 아니라, **블로그 운영 이력을 저장하고 비교하는 로컬 퍼스트 전략 대시보드**라는 점이 핵심입니다.

다음 Step에서는 실제로 이 요구를 받쳐 주는 **SQLite 스키마와 블로그 수집기 구조**를 어떻게 만들었는지 정리합니다.
