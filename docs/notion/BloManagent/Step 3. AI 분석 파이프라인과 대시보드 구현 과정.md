# AI 분석 파이프라인과 대시보드 구현 과정

> 메타 설명  
> BloManagent에서 Google AI Studio, OpenAI, Ollama를 선택형으로 연결하고, 포스트 분석부터 주간 요약, EBI 계산, 추천, 대시보드 UI까지 어떻게 구현했는지 설명합니다.

> 추천 슬러그  
> `blomanagent-ai-analysis-dashboard-implementation`

> 핵심 키워드  
> AI 블로그 분석, Google AI Studio 블로그 분석, Ollama 로컬 LLM 대시보드, 블로그 EBI 계산

## 들어가며

수집과 저장 구조가 준비되면 그다음은 실제 분석입니다.  
이번 프로젝트는 단순히 글을 요약하는 것이 아니라, **포스트 분석 -> 주간 리포트 -> 추천 생성 -> EBI 계산 -> 대시보드 반영**까지 하나의 흐름으로 연결하는 것이 중요했습니다.

## 목차

1. 왜 AI 제공자 추상화가 필요했는가
2. 포스트 분석 파이프라인
3. 주간 요약과 추천 생성
4. EBI 계산 방식
5. 대시보드 UI 구성

## 왜 AI 제공자 추상화가 필요했는가

사용자는 Google AI Studio를 기본으로 원했지만, 환경에 따라 OpenAI나 Ollama도 선택할 수 있어야 했습니다.  
그래서 비즈니스 로직이 특정 API에 묶이지 않도록 `AIProvider` 인터페이스를 두고 구현체를 분리했습니다.

이렇게 하면 다음 장점이 있습니다.

- 제공자 변경 시 핵심 분석 흐름 수정이 줄어듭니다
- fallback 로직을 넣기 쉽습니다
- 로컬 LLM 전환 실험이 쉬워집니다

## 포스트 분석 파이프라인

포스트 분석은 다음 순서로 진행합니다.

1. 분석 런 생성
2. 분석 범위에 맞는 포스트 선택
3. 본문 길이 제한 적용
4. 제공자 호출
5. JSON 스키마 검증
6. DB 저장

### 왜 JSON 스키마 검증을 넣었는가

LLM 응답은 종종 형식이 흔들립니다.  
그래서 응답을 바로 저장하지 않고 Zod 스키마로 검증한 뒤에만 저장하도록 설계했습니다.

이 덕분에 아래 필드들이 안정적으로 유지됩니다.

- summary
- topicLabels
- strengths
- weaknesses
- improvements
- titleStrength
- structureScore
- seoPotentialScore

## 주간 요약과 추천 생성

포스트 분석만으로는 블로그 전체 방향을 설명하기 어렵습니다.  
그래서 포스트 단위 분석이 끝난 뒤 다시 한 번 묶어서 주간 리포트를 생성합니다.

### 주간 요약에서 보는 것

- 어떤 주제가 반복되는지
- 빠진 주제는 무엇인지
- 다음 주에 보강할 토픽은 무엇인지
- 지금 가장 먼저 수정해야 할 액션은 무엇인지

### 추천 생성에서 보는 것

- 즉시 수정할 제목 전략
- 콘텐츠 믹스 조정
- SEO 관점 보완
- 카테고리 정리 방향

즉, 포스트 분석이 "각 글의 진단서"라면, 주간 리포트는 "블로그 전체 운영 회의록"에 가깝습니다.

## EBI 계산 방식

외부 공식 점수가 없기 때문에 내부 점수 체계인 **EBI(Estimated Blog Index)** 를 계산하도록 만들었습니다.

### EBI에 반영한 요소

- 발행 일관성
- 주제 다양성
- 콘텐츠 품질
- 구조 점수
- 실용성
- SEO 잠재력
- 독자 적합성
- 최신성
- 참여 지표

### 왜 EBI가 필요한가

사용자는 수십 개 지표보다, 먼저 **한눈에 파악할 수 있는 요약 지표**를 원합니다.  
EBI는 그 역할을 맡고, 세부 점수는 상세 페이지에서 확인하도록 분리했습니다.

## 대시보드 UI 구성

UI는 복잡한 분석 도구보다 **바로 상태를 파악하는 운영 패널**에 가깝게 구성했습니다.

### 대시보드

- 블로그 수
- 완료된 분석 런
- 최신 추천 수
- 블로그 카드별 최신 EBI와 직전 대비 변화

### 블로그 목록

- 등록된 블로그 확인
- Discover 버튼
- Analyze Now 버튼
- 삭제

### 블로그 상세

- 점수 이력
- 추천 목록
- 포스트 목록
- 토픽 라벨 확인

### 로그

- 분석 진행 상태 확인
- 각 단계 이벤트 추적

## 휴리스틱 fallback을 둔 이유

현실적으로 API Key가 없거나 외부 제공자 호출이 실패할 때도 있습니다.  
그래서 Phase 1에서는 완전 실패 대신 **결정론적 휴리스틱 분석 fallback** 을 넣었습니다.

이 선택의 장점은 다음과 같습니다.

- 앱 데모가 막히지 않습니다
- UI와 저장 흐름을 끝까지 검증할 수 있습니다
- 나중에 실제 모델 품질만 높이면 전체 구조는 유지됩니다

## 정리

Step 3의 핵심은 분석 품질보다 먼저 **분석 구조를 무너지지 않게 만드는 것**이었습니다.  
제공자 선택, JSON 검증, 비용 추적, 런 로그, EBI 계산, 대시보드 반영까지 하나의 흐름으로 묶으면서, 이후 확장 가능한 기반을 만들었습니다.

다음 Step에서는 테스트, 빌드, Windows 실행 방식, 그리고 향후 확장 전략을 정리합니다.
