## 테스트 전략

Phase 1에서 모든 크롤링 케이스를 E2E로 다루기는 어렵기 때문에, 우선 깨지기 쉬운 핵심 지점을 단위 테스트로 잡았습니다.

### 현재 테스트한 것

- URL 정규화
- 플랫폼 감지
- EBI 계산

### 왜 이 테스트부터 넣었는가

이 세 부분은 전체 구조에서 가장 자주 재사용되고, 한 번 깨지면 여러 화면과 저장 흐름이 동시에 흔들립니다.

즉, 지금 단계에서는 "가장 싼 비용으로 가장 큰 리스크를 막는 테스트"를 먼저 둔 것입니다.

## 빌드 구조 정리

프로젝트는 npm workspace 기준으로 나눴습니다.

- `packages/shared`: 공용 타입과 스키마
- `apps/server`: Fastify 서버와 러너
- `apps/web`: React UI

### 빌드 스크립트

- `npm run dev`
- `npm run build`
- `npm run start`

여기서 중요한 점은 **개발 모드와 운영 모드의 책임을 명확히 분리**한 것입니다.

### 서버 빌드에서 tsup을 쓴 이유

Node ESM 환경에서 상대 경로와 확장자 문제를 단순하게 처리하기 위해 서버는 번들 방식으로 정리했습니다.  
이렇게 하면 `npm run start`가 Windows에서도 더 안정적으로 동작합니다.

## Windows 실행 방식

이번 프로젝트는 Windows에서 쉽게 실행되는 것이 중요했습니다.  
그래서 `start.bat`를 추가해 다음 흐름을 자동화했습니다.

1. data 폴더 생성
2. `npm install` 확인
3. 빌드
4. 실행

이 구조 덕분에 개발자가 아니어도 비교적 쉽게 앱을 켤 수 있습니다.

## README와 문서화 전략

README는 단순 설치 안내가 아니라 **검색 유입과 프로젝트 이해를 동시에 잡는 문서**로 설계했습니다.

### README에 넣은 핵심 정보

- 프로젝트가 무엇인지
- 왜 만들었는지
- 어떤 문제를 해결하는지
- 어떤 블로그 플랫폼을 지원하는지
- 설치와 실행 방법
- 실제 사용 흐름

이렇게 해야 GitHub 저장소 검색과 Google 색인 양쪽에서 주제가 명확해집니다.

## 향후 확장 로드맵

### 1. 수집기 강화

- 티스토리 스킨별 셀렉터 보강
- 네이버 블로그 iframe/동적 구조 대응 강화
- Generic 수집기 정확도 개선

### 2. 리포트 강화

- 기간별 비교 전용 화면
- 월간/분기 리포트
- 토픽 겹침과 공백 시각화

### 3. 패키징 강화

- Electron 데스크톱 앱
- Tauri 기반 경량 exe
- 자동 업데이트 전략 검토

### 4. AI 전략 고도화

- Ollama 중심 로컬 분석 비중 확대
- OpenAI를 최종 검수 레이어로 분리
- 비용 제한 정책 강화

## 마무리

BloManagent를 만들면서 가장 중요하게 본 것은 "많이 하는 것"이 아니라 **계속 확장할 수 있게 시작하는 것**이었습니다.  
이번 Phase 1은 블로그 수집, 저장, 분석, 리포트, 실행 구조를 하나의 선으로 연결한 첫 번째 기반입니다.

이제 다음 단계에서는 더 강한 파서, 더 깊은 비교 리포트, 더 안정적인 패키징을 붙여서 실제 운영 도구 수준으로 끌어올릴 수 있습니다.
