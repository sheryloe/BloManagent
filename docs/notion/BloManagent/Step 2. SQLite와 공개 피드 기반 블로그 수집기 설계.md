# Step 2. SQLite와 공개 피드 기반 블로그 수집기 설계

## SEO 패키지

- SEO 제목: SQLite와 공개 피드 기반으로 블로그 수집기를 설계한 방법
- 슬러그: `sqlite-public-feed-blog-crawler-design`
- 메타 설명: BloManagent에서 SQLite 스키마를 어떻게 설계했고, RSS·sitemap·wp-json·main fallback 순서의 블로그 수집기를 왜 그렇게 구성했는지 정리합니다.
- 핵심 키워드: SQLite 블로그 수집기, RSS sitemap 수집기, 워드프레스 wp-json 수집
- 보조 키워드: 티스토리 RSS, 블로그 수집 아키텍처, 공개 블로그 크롤러
- 추천 태그: `SQLite`, `RSS`, `sitemap`, `crawler`, `WordPress`

## 도입

기획이 끝나면 바로 부딪히는 문제가 있습니다.  
“메인 URL 하나로 시작해서 실제 글 목록을 어떻게 안정적으로 찾을 것인가”입니다.

블로그 플랫폼은 제각각입니다.

- 티스토리는 RSS와 sitemap이 있지만, 스킨별 편차가 큽니다.
- 워드프레스는 `wp-json`이 강력하지만, 모든 사이트가 열어두는 것은 아닙니다.
- 블로그스팟은 공개 피드 구조가 비교적 단순하지만, 커스텀 도메인이 섞일 수 있습니다.

이런 상황에서 BloManagent는 화려한 수집기보다 “공개 구조를 최대한 우선 사용하고, fallback은 늦게 쓰는 방식”을 선택했습니다.

## 이 글에서 다루는 내용

1. 왜 SQLite를 골랐는가
2. posts와 분석 이력을 어떻게 저장했는가
3. discovery 순서를 왜 RSS → sitemap → wp-json → main으로 정했는가
4. 플랫폼별 어댑터를 왜 분리했는가

## 왜 SQLite였나

BloManagent는 서버가 항상 떠 있는 SaaS보다 로컬 워크플로우에 가까운 프로젝트입니다.  
그래서 데이터 저장소도 무겁지 않아야 했습니다.

SQLite를 선택한 이유는 단순합니다.

- 설치가 쉽다
- 파일 하나로 백업이 가능하다
- Windows 환경에서 다루기 편하다
- 로컬 데스크톱 앱으로 확장하기 쉽다

무엇보다 “사용자가 직접 실행하는 도구”라는 성격과 잘 맞았습니다.

## 어떤 테이블이 필요했나

이 프로젝트는 단순히 글 목록만 저장하면 끝나는 구조가 아닙니다.  
시간이 지나면서 다시 분석하고, 비교하고, 추천을 쌓아야 합니다.

그래서 아래 테이블이 핵심이 되었습니다.

### `blogs`

- 블로그 이름
- 메인 URL
- 플랫폼
- RSS URL override
- 활성 상태

### `posts`

- 실제 글 URL
- 제목
- 발행일
- 카테고리
- 태그
- 원문 HTML
- 정제 본문
- `content_hash`

여기서 중요한 필드는 `content_hash` 입니다.  
같은 URL이라도 글이 수정되면 해시가 바뀌고, 그때만 업데이트를 수행할 수 있습니다.

### `post_engagement_snapshots`

댓글 수, 공감 수, 조회 수처럼 시간이 지나며 달라지는 값은 별도 이력으로 분리했습니다.

### `analysis_runs`, `post_analyses`, `weekly_reports`

분석 실행, 글별 점수, 실행 단위 요약을 나눠서 저장해야 나중에 비교가 가능합니다.

## 수집 순서를 왜 이렇게 정했나

BloManagent의 discovery 기본 흐름은 아래와 같습니다.

1. RSS / Atom
2. sitemap
3. WordPress `wp-json`
4. main 링크 fallback

이 순서에는 이유가 있습니다.

### RSS를 먼저 보는 이유

RSS는 글 중심으로 잘 정리된 구조를 제공하는 경우가 많습니다.  
HTML을 긁는 것보다 안정적이고, 파싱 비용도 낮습니다.

### sitemap을 두 번째로 두는 이유

sitemap은 과거 글을 보완하기 좋습니다.  
RSS가 최근 글 위주로만 제공될 때, sitemap이 히스토리를 채워줍니다.

### wp-json은 워드프레스 전용 승급 포인트

워드프레스 사이트라면 `wp-json`이 가장 신뢰도 높은 공개 데이터 소스가 될 수 있습니다.  
그래서 플랫폼 판별 단계에서 워드프레스 여부를 먼저 확인하고, 가능한 경우 REST API를 적극 활용합니다.

### main fallback을 가장 늦게 두는 이유

메인 링크 추출은 마지막 수단입니다.  
사이트 구조와 스킨에 따라 category, tag, archive가 섞일 수 있기 때문입니다.

즉, 기본 철학은 이렇습니다.

> 구조화된 공개 데이터가 있으면 그것을 먼저 믿고, HTML 추출은 가능한 한 늦게 쓴다.

## 플랫폼별 어댑터를 왜 분리했나

플랫폼마다 HTML 구조와 글 URL 규칙이 다르기 때문에, 공통 파이프라인 하나로 전부 처리하면 예외 처리가 늘어납니다.

그래서 `BlogPlatformAdapter` 개념으로 분리했습니다.

- `isPostUrl()`
- `discoverPosts()`
- `fetchPost()`
- `extractEngagement()`

이렇게 인터페이스를 통일해두면, 공통 흐름은 유지하면서도 티스토리와 워드프레스처럼 성격이 다른 플랫폼을 유연하게 처리할 수 있습니다.

## 구현하면서 가장 중요했던 선택

수집기는 “많이 긁는 것”보다 “덜 틀리게 가져오는 것”이 더 중요했습니다.

특히 티스토리는 글처럼 보이는 URL이 많기 때문에,

- URL 규칙 필터
- article marker 확인
- verified / excluded 상태 관리

같은 방어 로직이 반드시 필요했습니다.

이 설계가 있었기 때문에 이후 Step 5에서 strict verified-post discovery로 자연스럽게 확장할 수 있었습니다.

## 마무리

Step 2에서 만든 것은 단순한 크롤러가 아니었습니다.  
메인 URL 하나로 시작해도 신뢰할 수 있는 글 목록을 만들고, 그 결과를 나중에 다시 분석 가능한 형태로 보관하는 데이터 기반을 만든 단계였습니다.
