## 왜 SQLite를 선택했는가

로컬 앱에서 가장 중요한 것은 설치와 운영의 단순성입니다.  
SQLite는 별도 DB 서버가 필요 없고, 파일 하나로 데이터 백업이 가능하며, Windows 로컬 실행과도 잘 맞습니다.

또한 이번 프로젝트는 다음 데이터를 모두 저장해야 했습니다.

- 블로그 메인 URL
- 발견된 포스트 URL
- 정제된 본문
- 참여 지표 스냅샷
- 분석 런 기록
- 포스트별 분석 결과
- 주간 리포트
- 추천 기록
- 비용 로그

즉, 단순 캐시가 아니라 **히스토리를 누적하는 운영 DB**가 필요했습니다.

## 핵심 테이블 설계

이번 Phase 1에서는 다음 테이블을 중심으로 구현했습니다.

### blogs

블로그의 기본 정보가 들어갑니다.

- 블로그 이름
- 메인 URL
- 플랫폼
- RSS URL
- 활성화 여부

### posts

발견한 포스트를 저장합니다.

- 포스트 URL
- 제목
- 발행일
- 카테고리
- 원문 HTML
- 정제 본문
- 내용 해시

여기서 **content_hash** 를 저장한 이유는 변경 감지 때문입니다.  
같은 URL이라도 본문이 바뀌었는지 바로 판단할 수 있어 재수집 정책을 단순화할 수 있습니다.

### post_engagement_snapshots

댓글 수, 좋아요, 공감 수, 조회수 같은 지표는 시간이 지나며 바뀝니다.  
그래서 포스트 테이블에 덮어쓰는 대신 별도 스냅샷 테이블로 저장했습니다.

### analysis_runs / post_analyses / weekly_reports

분석 실행 자체를 런 단위로 저장하고, 그 안에 포스트 분석과 주간 리포트를 연결했습니다.  
이 구조 덕분에 "이번 주 vs 지난달" 같은 비교가 가능해집니다.

## 포스트 수집 순서

수집은 다음 우선순위로 설계했습니다.

1. RSS / Atom
2. sitemap
3. 메인 페이지 링크 추출
4. 플랫폼별 fallback

이 순서를 택한 이유는 명확합니다.

- RSS와 sitemap은 구조가 안정적이고 빠릅니다
- HTML 스크래핑은 사이트마다 흔들릴 수 있습니다
- fallback은 꼭 필요하지만 항상 기본이 되면 유지보수 비용이 커집니다

즉, **안정적인 구조 먼저, HTML 파싱은 나중**이라는 원칙으로 갔습니다.

## 플랫폼별 수집기 분리 방식

블로그 플랫폼마다 HTML 구조가 다르기 때문에 공통 파이프라인만으로는 한계가 있습니다.  
그래서 `BlogPlatformAdapter` 형태로 수집기 계약을 분리했습니다.

### Blogger / Blogspot

- 상대적으로 구조가 단순합니다
- 연/월 기반 URL 패턴이 명확한 편입니다
- RSS 접근도 쉬운 편입니다

### Tistory

- 스킨에 따라 HTML 구조가 달라질 수 있습니다
- RSS가 자주 제공됩니다
- 카테고리/태그 선택자가 일정하지 않은 경우가 있습니다

### Naver Blog

- iframe 구조나 렌더링 이슈가 있어 예외가 많습니다
- 정적 HTML로 충분하지 않을 때가 있습니다
- 그래서 HTTP fetch가 실패하거나 구조가 이상하면 Playwright fallback을 고려했습니다

### Generic

- 워드프레스나 커스텀 블로그처럼 특정 플랫폼으로 분류되지 않는 사이트를 위한 범용 파서입니다

## 변경 감지와 스냅샷 저장

이번 구현에서 중요한 포인트는 **덮어쓰기 대신 누적 저장**입니다.

### 포스트 변경 감지

포스트를 다시 가져왔을 때 정제 본문을 해시 처리하고, 기존 해시와 비교합니다.

- 해시가 다르면 내용이 바뀐 것
- 해시가 같으면 재분석 필요도가 낮음

### 참여 지표 스냅샷

댓글 수와 공감 수는 시간이 지나면 달라질 수 있기 때문에 별도 스냅샷으로 저장합니다.

### 왜 이 구조가 중요한가

블로그 운영은 시계열 데이터입니다.  
오늘의 점수만 보면 의미가 약하고, **3개월 전보다 어떻게 변했는지**를 봐야 전략이 나옵니다.
