- 핵심 키워드: 분석 대시보드 UX, GitHub Pages 제품 문서, README 재구성

- 추천 태그: `Dashboard`, `UX`, `README`, `GitHub Pages`, `Analytics`

## 도입

수집기와 점수 알고리즘을 붙이고 나면 바로 쓸 수 있을 것 같지만 실제로는 그렇지 않았습니다.
데이터가 있어도 화면이 어수선하면 사용자는 무엇을 먼저 해야 하는지 모릅니다.
특히 BloManagent는 “한 번 설치하고 가끔 열어서 바로 판단하는” 워크스페이스이기 때문에, 첫 화면이 복잡하면 기능보다 피로감이 먼저 옵니다.

그래서 Step 4에서는 기능을 더 만드는 대신, 이미 있는 기능이 더 빨리 이해되고 바로 행동으로 이어지도록 대시보드와 문서를 다시 설계했습니다.

## 왜 평균 점수보다 바로 할 일을 먼저 보여줘야 했나

초기 화면은 평균 점수와 실행 로그 비중이 높았습니다.
하지만 사용자가 실제로 보고 싶은 것은 “이 블로그가 72점입니다”가 아니라 “지금 어떤 글부터 손봐야 합니까”였습니다.

그래서 대시보드는 아래 순서로 재배치했습니다.

1. 최신 분석 브리핑
2. 위험 글 수와 등급 분포
3. 반복 병목 상위 항목
4. Best/Worst 게시글
5. 즉시 수정할 액션

이 구조로 바꾸면 화면이 결과 발표용이 아니라 작업 관제판처럼 보입니다.
BloManagent가 지향하는 사용 맥락은 정확히 그쪽입니다.

## 휘발성 워크스페이스에 리포트 목록이 왜 안 맞았나

처음에는 분석 리포트 목록을 남겨두는 구성이 있었지만, 이 프로젝트는 장기 히스토리 브라우저보다 “그때그때 다시 돌리는 작업대”에 가깝습니다.
그래서 최신 스냅샷만 진하게 보여주고, 이전 리포트 목록은 과감히 걷어냈습니다.

이 결정으로 얻은 장점은 분명했습니다.

- 사용자가 지금 봐야 할 리포트가 하나로 고정된다
- 화면 정보량이 줄어든다
- 휘발성 워크스페이스라는 제품 성격과 맞아떨어진다

결국 리포트 목록 제거는 기능 축소가 아니라 제품 성격을 명확하게 만드는 정리였습니다.

## README와 GitHub Pages를 왜 따로 다듬었나

GitHub 저장소 첫 화면과 GitHub Pages는 역할이 다릅니다.

- README는 저장소 진입 후 30초 안에 프로젝트를 이해시키는 문서
- GitHub Pages는 구조와 사례를 시각적으로 설명하는 랜딩 문서

그래서 README에는 프로젝트 목적, 지원 플랫폼, 실행 방법, 등급 체계, strict verified-post discovery 같은 핵심 구조를 바로 배치했습니다.
반면 GitHub Pages는 네이비 계열 BI 리포트 톤으로, 카드형 지표와 흐름도, 사례 보드 중심의 문서형 랜딩으로 구성했습니다.

이렇게 분리해야 코드 저장소와 제품 소개 페이지가 서로 같은 말을 하면서도 각자 역할을 수행합니다.

## 용어를 정리하지 않으면 문서와 화면이 서로 충돌한다

UI를 손볼 때 같이 정리한 것이 용어였습니다.

- `EBI` 제거
- `qualityScore`와 `S~F 등급` 중심으로 통일
- `verified post count` 의미 명확화
- `algorithm이 등급 계산, AI는 서술 보강` 원칙 고정

이 정리가 없으면 도움말, 설정 화면, 리포트, README가 서로 다른 제품처럼 읽힙니다.
화면 디자인보다 용어 정리가 먼저였던 이유가 여기에 있습니다.

## 결과적으로 어떤 화면이 되었나

최종적으로 BloManagent의 화면은 일반 블로그 툴보다는 경량 BI 대시보드에 가까워졌습니다.
좌측 네비게이션, 상단 브리핑, KPI 카드, Best/Worst 비교, 등급 분포, 반복 병목 요약을 한 화면 안에서 읽게 했고, GitHub Pages도 같은 톤으로 맞췄습니다.

이런 형태는 예뻐 보이기 위한 선택이 아니라, 사용자가 분석 결과를 “읽는 것”이 아니라 “판단하는 것”으로 받아들이게 하기 위한 선택입니다.
