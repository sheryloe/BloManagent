# Step 4. 대시보드와 GitHub Pages 문서를 제품처럼 다듬는 과정

## SEO 패키지

- SEO 제목: 대시보드와 GitHub Pages 문서를 제품처럼 다듬은 과정
- 슬러그: `dashboard-github-pages-product-documentation`
- 메타 설명: BloManagent에서 게시글 진단 결과를 실제 행동으로 이어지게 하기 위해 대시보드, 도움말, README, GitHub Pages를 어떤 방향으로 재구성했는지 정리합니다.
- 핵심 키워드: 대시보드 UX 개선, GitHub Pages 문서 디자인, 개발 프로젝트 README 구성
- 보조 키워드: 제품형 문서, 오픈소스 랜딩페이지, 사용자 행동 중심 UI
- 추천 태그: `Dashboard`, `README`, `GitHub Pages`, `UX`, `문서디자인`

## 도입

기능을 만드는 것과 사람들이 실제로 쓰게 만드는 것은 다른 문제였습니다.

BloManagent는 블로그 운영자가 직접 여는 워크스페이스를 목표로 했기 때문에, 데이터가 좋아도 화면이 어수선하면 의미가 없었습니다.  
특히 아래 문제가 분명했습니다.

- 평균 점수만 보면 무엇을 먼저 해야 할지 잘 안 보인다
- 도움말이 딱딱하면 처음 들어온 사람이 이탈한다
- README가 설치 설명만 있으면 프로젝트의 목적이 안 보인다

그래서 Step 4에서는 “보여주는 방식”을 다시 설계했습니다.

## 이 글에서 다루는 내용

1. 왜 대시보드를 글 단위 우선으로 바꿨는가
2. 도움말 페이지는 어떤 질문에 답하도록 구성했는가
3. GitHub Pages는 왜 제품 랜딩처럼 만들었는가
4. README는 어떻게 깃헙 첫 화면 역할을 하게 했는가

## 왜 대시보드를 글 단위 우선으로 바꿨나

초기 대시보드는 블로그 평균과 실행 로그 위주였습니다.  
하지만 사용자가 실제로 원하는 것은 “다음 액션”이었습니다.

그래서 아래 항목을 앞쪽으로 끌어왔습니다.

- 최신 qualityScore
- 분석된 글 수
- 주의 글 수
- 반복 이슈 상위 3개
- 최근 분석 글 목록
- 각 글의 상위 개선 포인트

이렇게 바꾸면 대시보드가 보고서가 아니라 작업 큐처럼 보입니다.

## 도움말 페이지는 무엇을 설명해야 했나

도움말 페이지는 단순 기능 목록이 아니라, 처음 보는 사람이 막히는 지점을 먼저 풀어야 했습니다.

그래서 설명 순서를 아래처럼 잡았습니다.

1. 등록은 어떻게 하는가
2. discover는 무엇을 찾는가
3. qualityScore는 무엇을 의미하는가
4. 왜 티스토리 글 수가 다르게 보일 수 있는가
5. 정책상 주의할 플랫폼은 무엇인가

즉, 개발자 입장에서 중요한 설명보다 사용자 입장에서 막히는 질문을 먼저 배치했습니다.

## GitHub Pages를 왜 제품 랜딩처럼 만들었나

오픈소스 프로젝트는 GitHub README만 봐도 되지 않을까 싶었습니다.  
그런데 실제로는 그렇지 않았습니다.

프로젝트가 무엇을 해결하는지, 어떤 흐름으로 동작하는지, 왜 이런 설계를 했는지 빠르게 이해시키려면 별도의 랜딩이 훨씬 유리합니다.

그래서 GitHub Pages에는 아래 역할을 맡겼습니다.

- 제품 소개
- 구조 요약
- 사례 기반 설명
- Step 시리즈 안내
- 도움말 진입점

디자인도 평범한 문서 톤 대신, 약간은 엔지니어링 보드처럼 보이게 잡았습니다.  
어두운 배경, 밝은 패널, 강한 타이포, 숫자 카드와 시스템 맵 구성이 그 방향에서 나온 선택입니다.

## README는 어떤 역할을 해야 하나

GitHub 저장소에 처음 들어온 사람은 대부분 README부터 봅니다.  
그래서 README는 설치 문서이면서 동시에 프로젝트 소개 문서여야 했습니다.

이번에 README를 다시 정리하면서 아래 항목을 중심에 두었습니다.

- 이 프로젝트가 무엇인지
- 왜 만들었는지
- qualityScore가 무엇인지
- 티스토리 strict verified-post discovery가 왜 필요한지
- 어떤 플랫폼을 지원하는지
- 바로 실행하려면 무엇을 해야 하는지

결국 README는 “코드를 보기 전 30초 안에 프로젝트를 이해시키는 문서”가 되어야 했습니다.

## 문서와 UI를 같이 다듬어야 했던 이유

제품 화면과 문서가 서로 다른 말을 하면 신뢰가 떨어집니다.  
그래서 용어도 같이 정리했습니다.

- EBI 제거
- qualityScore 중심 용어 통일
- algorithm-first 구조 강조
- verified post count 의미 명확화

이렇게 해야 대시보드, README, GitHub Pages, 도움말이 같은 제품처럼 읽힙니다.
