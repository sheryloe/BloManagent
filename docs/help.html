Quick start

처음 보는 사람도 바로 실행할 수 있게 정리했습니다.

BloManagent는 기본적으로 공개 블로그 URL만 받습니다. 설치 후에는 블로그 등록, discover, analyze 순서로 바로 결과를 볼 수 있습니다.

01

블로그 등록

Tistory, Blogger, WordPress 메인 URL을 등록합니다. RSS URL은 고급 옵션입니다.

02

Discover

RSS, sitemap, wp-json, main fallback 순서로 공개 글을 찾습니다.

03

Analyze Now

algorithm 엔진이 게시글별 qualityScore와 개선 포인트를 계산합니다.

04

우선순위 확인

낮은 점수 글과 반복 이슈를 대시보드에서 바로 확인합니다.

qualityScore

대표 점수는 블로그 평균보다 게시글 단위 qualityScore입니다.

EBI 같은 종합 지표는 제거했고, 지금은 각 글을 설명 가능한 기준으로 먼저 진단합니다. AI를 켜더라도 점수 자체는 algorithm이 고정 계산합니다.

A

headlineScore

제목 선명도와 첫 문단 훅이 충분한지 봅니다.

B

readabilityScore

소제목, 문단 길이, 전개 구조를 기준으로 읽기 편한지 봅니다.

C

valueScore

실행 가능한 정보, 예시, 체크리스트가 충분한지 확인합니다.

D

originalityScore

차별점, 경험 기반 설명, 비교 관점이 살아 있는지 점검합니다.

E

searchFitScore

검색 의도 정렬, 질문형 소제목, SEO 잠재력을 반영합니다.

80 이상: excellent
65-79: solid
50-64: watch
49 이하: needs-work

Tistory strict mode

왜 어떤 티스토리는 글이 12개쯤인데 56개처럼 잡히는가

티스토리 sitemap과 메인 링크에는 실제 글 외에 category, tag, archive, guestbook이 섞일 수 있습니다. BloManagent는 이를 막기 위해 verified post만 집계하는 strict discovery를 적용했습니다.

Gate 1

URL 규칙

/{숫자}, /entry/...만 허용하고 category, tag, archive 경로는 차단합니다.

Gate 2

실제 페이지 fetch

RSS와 sitemap에서 찾은 URL도 실제 페이지를 다시 가져와 article marker를 확인합니다.

Gate 3

verified / excluded 상태

애매한 URL은 삭제하지 않고 내부적으로 excluded로 남기되, 화면 집계와 분석 대상에서는 제외합니다.

Source order RSS → sitemap → main fallback

main 링크는 RSS와 sitemap이 둘 다 0건일 때만 최후 fallback으로 사용합니다.

Storybeing 14 verified posts

rss 10 / sitemap 4 / main 0 / wp-json 0으로 재검증되었습니다.

Policy notes

공개 데이터만 다루고, 정책 리스크는 보수적으로 봅니다.

01

공개 페이지만 요청

로그인 쿠키, 비공개 글, 관리자 통계는 수집하지 않습니다.

02

Naver는 opt-in

정책 리스크 때문에 기본값은 allowNaverPublicCrawl=false 입니다.

03

LLM은 선택형

OpenAI, Google, Ollama는 설명 문장 보강 용도이며, 기본 점수 계산은 algorithm이 맡습니다.

Step series

노션과 티스토리용 Step 1~5 글감도 같이 정리했습니다.

아래 Step 문서는 구현기이면서 동시에 검색 유입을 노릴 수 있는 블로그형 구조로 작성했습니다. 현재는 docs/notion/BloManagent 폴더에 원고를 두었고, 노션 페이지 권한이 열리면 바로 하위 페이지로 생성할 수 있습니다.

자동 생성 스크립트를 쓰려면 대상 페이지를 integration donggri에 먼저 공유해야 합니다.

Step 1

메인 URL 기반 기획

문제 정의, 로컬 중심 워크플로우, 왜 URL-only가 중요한지 다룹니다.

Step 2

SQLite와 공개 피드 수집

RSS, sitemap, wp-json, HTML fallback 구조를 다룹니다.

Step 3

LLM 없이 점수 계산

qualityScore를 설명 가능한 규칙 기반으로 계산하는 과정을 다룹니다.

Step 4

대시보드와 문서 UX

README, GitHub Pages, 도움말을 제품 문서처럼 재구성한 이유를 설명합니다.

Step 5

strict verified-post discovery

티스토리 오탐 문제를 어떻게 설계적으로 막았는지 정리합니다.
