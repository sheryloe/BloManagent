Quick Start

처음 보는 사람도 바로 따라 하는 실행 순서

BloManagent는 메인 URL만 넣어도 되고, 게시글 URL을 넣어도 됩니다. 필요한 만큼만 수집하고, 결과를 본 뒤 초기화하는 휘발성 워크스페이스 흐름을 기본으로 삼고 있습니다.

블로그 주소 입력 자동 수집 분석 시작 리포트 확인
입력 블로그 / 게시글 주소

게시글 주소를 넣으면 가능한 범위에서 블로그 기준으로 정규화합니다.

분석 algorithm 기본

등급 계산은 algorithm이 맡고, AI는 필요할 때만 서술 보강용으로 씁니다.

Runbook

실제 사용 순서

01

주소 입력

티스토리, 워드프레스, 블로거 메인 주소 또는 개별 게시글 주소를 입력합니다.

02

자동 수집

RSS, sitemap, wp-json, HTML을 이용해 공개 게시글 후보를 모으고 검증합니다.

03

분석 시작

게시글별 등급, 강한 신호, 약한 신호, 개선 제안, raw metric을 생성합니다.

04

리포트 확인

Best/Worst 글, 분포, 병목, 우선 실행 항목을 보고 필요한 수정만 바로 정리합니다.

Grade System

화면은 S부터 F까지, 내부 계산은 qualityScore 기반

사용자 입장에서는 숫자보다 등급이 더 빠르게 읽히기 때문에, UI와 리포트는 등급 중심으로 바꿨습니다. 다만 내부적으로는 qualityScore를 계산한 뒤 S~F로 변환합니다.

상위 5개 축

  • headlineScore
  • readabilityScore
  • valueScore
  • originalityScore
  • searchFitScore

화면에 보이는 정보

  • S~F 등급
  • Best / Worst 글
  • 강한 신호 / 약한 신호
  • 개선 제안

등급 구간

  • S: 90+
  • A: 80+
  • B: 65+
  • C: 55+
  • D: 45+
  • F: 45 미만

Tistory Rule

티스토리에서 글 수가 부풀어 보이는 문제를 막는 방식

티스토리는 sitemap과 메인 링크에 비게시글 URL이 섞일 수 있습니다. 그래서 “홈에서는 12개쯤 보이는데 앱에서는 56개로 인식한다” 같은 문제가 발생할 수 있었습니다. 현재는 strict verified-post discovery로 이 문제를 구조적으로 막았습니다.

Gate 1

URL 규칙

/{숫자}, /entry/...만 통과시키고 category, tag, archive, guestbook 등은 제외합니다.

Gate 2

실제 페이지 확인

RSS와 sitemap에서 찾은 URL을 다시 fetch해 article marker가 있는지 확인합니다.

Gate 3

verified / excluded 분리

애매한 URL은 삭제하지 않고 excluded로 보관하되, 집계와 분석 대상에서는 빼줍니다.

storybeing.tistory.com 현재 기준

  • verified posts: 14
  • rss: 10
  • sitemap: 4
  • main: 0
  • wp-json: 0

Policy Scope

공개 범위만 다루고, 정책 리스크는 보수적으로 처리합니다.

항목 현재 정책 메모
비공개 글 수집 안 함 로그인이나 쿠키가 필요한 영역은 읽지 않습니다.
관리자 통계 수집 안 함 오너 전용 지표는 v1 범위에서 제외합니다.
Naver Blog 기본 비활성 allowNaverPublicCrawl=false가 기본값입니다.
AI 엔진 선택형 Ollama, OpenAI, Google은 설명 문장 보강용입니다.

Troubleshooting

자주 생기는 질문

글 수가 실제보다 많아 보여요

예전 false positive가 남았을 수 있습니다. 다시 수집을 실행하면 verified 기준으로 재분류됩니다.

점수 대신 왜 등급으로 보이나요

사용자가 해석하기 쉽도록 화면은 S~F 등급 중심으로 바꿨고, 내부 점수는 계속 계산됩니다.

AI 없이도 되나요

됩니다. 알고리즘만으로 수집, 분석, 리포트, 대시보드 흐름이 완성되도록 설계했습니다.
