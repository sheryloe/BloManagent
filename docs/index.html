Executive Summary

지금 버전은 “게시글별 진단”에 집중한 로컬 분석 도구입니다.

블로그 평균만 보는 구조가 아니라, 어떤 글이 가장 먼저 손봐야 하는지, 반복 병목이 무엇인지, 수집된 글 수가 왜 그렇게 계산됐는지를 한눈에 설명할 수 있는 방향으로 다시 설계했습니다.

입력 방식 블로그 주소 / 게시글 주소 게시글 주소를 넣으면 가능한 범위에서 블로그 기준으로 정규화합니다.
화면 표기 S ~ F 등급 내부는 qualityScore로 계산하지만 화면과 리포트는 등급 중심으로 보여줍니다.
수집 기준 Verified posts only 비게시글 URL은 저장하더라도 집계와 분석 대상에서는 제외합니다.
운영 방식 휘발성 워크스페이스 필요할 때 수집하고, 끝나면 초기화하는 흐름을 기본으로 설계했습니다.

Workflow

공개 데이터 수집부터 리포트 생성까지 4단계로 단순화했습니다.

01

Discovery

RSS, sitemap, WordPress wp-json, 메인 링크를 순서대로 확인하며 공개 게시글 후보를 수집합니다.

02

Verification

URL 규칙과 실제 페이지의 article marker를 함께 확인해 검증된 게시글만 통과시킵니다.

03

Scoring

제목·첫인상, 가독성, 정보 가치, 차별성, 검색 적합성 5축으로 게시글 단위 등급을 계산합니다.

04

Action

Best/Worst 글, 분포, 반복 병목, 다음 실행 항목을 리포트와 대시보드에서 바로 보여줍니다.

Tistory Case

왜 예전엔 56개처럼 보였고 지금은 14개인가

티스토리 sitemap과 메인 링크는 실제 글 외에 category, tag, archive, guestbook 같은 URL을 함께 노출할 수 있습니다. 지금 버전은 strict verified-post discovery를 적용해 글로 검증된 URL만 집계합니다.

Before

비게시글 URL이 섞일 수 있었습니다.

메인 링크 fallback이 너무 넓을 경우 category, tag, archive가 postCount에 포함될 수 있었습니다.

After

검증된 공개 게시글만 집계합니다.

허용 URL 규칙, 실제 페이지 fetch, article marker 확인을 모두 통과한 글만 verified 처리합니다.

Step Series

노션과 티스토리용 Step 시리즈도 함께 정리했습니다.

구현 이유, 데이터 구조, 알고리즘, UI 재설계, 티스토리 strict filtering까지 단계별로 옮겨 적을 수 있게 블로그형 원고를 별도로 관리하고 있습니다.

Step 1

메인 URL 기반 기획

왜 URL-only 흐름이 필요한지, 왜 로컬 워크스페이스가 맞는지 정리합니다.

Step 2

공개 필드 수집기 설계

RSS, sitemap, wp-json, HTML fallback 구조와 posts 저장 방식을 설명합니다.

Step 3

Algorithm 기반 진단

LLM 없이 게시글별 qualityScore와 S~F 등급을 계산하는 방식에 집중합니다.

Step 4

대시보드와 문서 UX

README, GitHub Pages, 앱 UI를 분석형 리포트 구조로 바꾼 이유를 다룹니다.

Step 5

strict verified-post discovery

티스토리 글 수가 왜 틀어지는지, 그 문제를 어떻게 구조적으로 막았는지 설명합니다.
