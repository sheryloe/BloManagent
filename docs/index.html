게시글 진단까지

BloManagent는 공개 블로그를 메인 URL 기준으로 수집하고, 게시글별 qualityScore와 개선 포인트를 계산하는 로컬 중심 분석 워크스페이스입니다. OAuth를 기본 조건으로 두지 않고, LLM 없이도 설명 가능한 진단 결과를 먼저 보여주는 흐름에 집중했습니다.

Why this exists

블로그 글쓰기보다 운영 진단이 더 어려웠습니다.

콘텐츠 도구는 많지만, 실제 운영자는 “어떤 글을 먼저 고쳐야 하는가”, “플랫폼이 달라도 같은 기준으로 볼 수 있는가”, “API 키 없이 공개 데이터만으로도 어느 정도 볼 수 있는가”를 먼저 궁금해합니다. BloManagent는 이 질문에서 시작했습니다.

01

메인 URL 중심

사용자는 RSS 주소나 관리자 API를 기억하지 않아도 됩니다. 메인 URL만 넣으면 공개 피드와 링크를 먼저 탐색합니다.

02

게시글 단위 진단

블로그 평균보다 지금 손봐야 할 글을 먼저 보여주도록, 게시글별 qualityScore와 개선 제안을 중심에 둡니다.

03

검증된 글만 집계

티스토리처럼 category, tag, archive가 섞이는 경우를 막기 위해 verified-post 기준으로 수집 글 수를 셉니다.

System map

공개 데이터 수집부터 점수 계산까지 흐름을 단순하게 유지합니다.

Step A

Discovery

RSS, sitemap, WordPress wp-json, 메인 링크를 순서대로 확인합니다.

Step B

Verification

URL 규칙과 실제 페이지 마커를 확인해 verified post만 통과시킵니다.

Step C

Scoring

headline, readability, value, originality, search fit 다섯 점수로 qualityScore를 계산합니다.

Step D

Action

낮은 점수 글, 반복 이슈, 다음 수정 우선순위를 대시보드에 바로 노출합니다.

headlineScore = avg(titleStrength, hookStrength)
valueScore = avg(informationDensityScore, practicalityScore)
qualityScore = round(avg(5 components))

Tistory case study

SEO article series

티스토리 게시용 Step 1~5 원고를 별도로 정리했습니다.

구현 과정, 설계 이유, 실수와 수정 포인트까지 단계별로 풀어낸 블로그형 원고입니다. 노션에 넣고 그대로 다듬어 티스토리에 올릴 수 있게 구성했습니다.

Step 1

기획과 문제 정의

왜 메인 URL 하나로 시작해야 했는지, 왜 로컬 중심 흐름이 맞는지 정리합니다.

Step 2

SQLite와 수집기 설계

공개 피드 기반 수집 순서와 posts/history 구조를 설명합니다.

Step 3

qualityScore 알고리즘

LLM 없이도 설득력 있는 게시글 점수를 어떻게 계산했는지 다룹니다.

Step 4

대시보드와 문서 UX

사용자가 실제로 보는 화면과 GitHub Pages 문서 톤을 맞춘 과정을 정리합니다.

Step 5

티스토리 strict filtering

sitemap 오탐을 막고 verified post만 집계하는 방식을 설명합니다.
